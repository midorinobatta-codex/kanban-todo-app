# FlowFocus 運用メモ

## 1. 毎日の使い方
1. 朝は Board の「今日」を開く
2. `今やる1件 + 次にやる2件` から最初の 1 手を選ぶ
3. Today で実行対象を確認し、Review で waiting 超過 / waiting 日付未設定 / 進行停滞 / 次アクション未設定 project を先に潰す
4. Project 単位で詰まっている場合は Project Detail の `今やる1件 + 次にやる2件` を確認する
5. 週次では Viewer で期限偏りと started_at / due_date 欠落を確認する

## 2. 止まり案件の見方
### 停滞として拾うもの
- waiting で回答予定日を超過
- waiting だが回答予定日が未設定
- doing のまま 3 日以上更新なし
- 未完了のまま due_date を超過
- project に次アクションがない

### 解釈の基本
- `回答予定日超過` は最優先で再確認・催促
- `待ち日付未設定` はまず日付を入れる
- `進行停滞` はやる / 待つ / 分解する の判断を入れる
- `期限超過` は今日ビューの上位候補として扱う
- `次アクション未設定` は project が止まっている合図として扱う

## 3. 入力を増やさない運用
- waiting へ送るときは `待ち＋日付` を優先し、モーダル入力を増やさない
- Review モードでは `完了` / `待ち＋日付` / `someday` / `todo・doingへ戻す` を最短で処理する
- 一括操作では日付未入力でも次営業日を自動提案して使う
- Board / Projects / Viewer の焦点表示は既存データから自動算出されるので、追加メモを入力しなくてよい
- 入力が必要なのは、仕事内容そのものを分解するときと待ちの回答予定日を明示したいときだけに絞る

## 4. 件数が増えたときの運用
- 一覧は初期件数だけ描画し、必要なときだけ `さらに N 件` を押す
- 長い列は全部を一度に見ようとせず、`今やる1件 + 次にやる2件` と `止まり案件` を起点に判断する
- 複数更新は一括操作を使い、D&D 連打は避ける

## 5. Export の使い分け
- CSV
  - Excel で確認したいとき
  - 配布・レビュー用
- JSON
  - 構造を保って保存したいとき
  - 後続処理へ渡したいとき

## 6. 履歴の扱い
- 履歴は localStorage 保存
- 共有監査ログではない
- サイトデータ削除で消える
- 不要になったら各画面の「履歴をクリア」で整理

## 7. リリース前チェック
- Board / Projects / Project Detail / Viewer の表示崩れがないか
- `今やる1件 + 次にやる2件` が上位候補を自然に拾えているか
- `止まり案件` の件数と実データが大きくズレていないか
- waiting 解除時に `waiting_response_date` が消えるか
- started_at が再上書きされていないか
- CSV が Excel で文字化けしないか
- `さらに N 件` で段階描画が壊れていないか


## デザイン調整方針
- 最上段は判断専用バーとして、画面名・件数・主操作だけを残す
- 「今やる1件」は最も目立つカードにして、次点候補と見た目を分ける
- 色の役割は固定する（赤=危険、黄=注意、青=現在地/主操作、灰=補足、緑=完了）
- カード内の情報順は タイトル → 理由タグ → 日付 → Project → 担当に寄せる
- 履歴・通知/警告・補助一覧などの二次情報は折りたたみ前提にする


## 最終デザイン磨き込み
- 主役 / 準主役 / 脇役 の強弱を整理
- タグは状態・理由・属性の3系統に寄せて表示を簡素化
- 止まり案件は危険順で先頭に出る前提
- 履歴や補助一覧は折りたたみを基本にして一覧の集中を維持


## PWA 運用
- PC では Chrome / Edge で `インストール` を押すとアプリのように起動できます。
- iPhone / iPad は Safari の共有メニューから `ホーム画面に追加` を使ってください。
- service worker 更新が反映されない場合は、アプリを閉じて再起動するか、ブラウザ側で再読み込みしてください。
